# Персональная лента «Мой участок» — дизайн

**Дата:** 2026-06-22
**Статус:** утверждён, готов к плану реализации
**Контекст:** пост-MVP. Пользователь попросил «персональную ленту с фото как основу для будущей социализации». Сейчас фото живут только пер-посадочно (внутри `PlantingInfoScreen`); сводной ленты и агрегирующего эндпоинта нет.

## 1. Цель и охват

Приватная **сводная лента** всех фото пользователя по всем посадкам + ключевые **вехи сезона**. Назначение — «дневник роста» как мотивация и **фундамент под будущую социализацию**.

**В объёме v1:** только личная приватная лента. Соц-функции (шеринг, профили, подписки, лайки) **не реализуются** — но форма данных и экран проектируются так, чтобы соц-слой добавился позже без переделки.

**Решения брейншторма:**
- Охват: *личная лента сейчас, соцфункции потом* (вариант A).
- Состав: *фото + вехи сезона* (вариант B).
- Размещение: вкладка *«Профиль / Мой участок»*, заменяет *«Инфо»* (вариант C).
- Компоновка: *профиль с вкладками* (вариант B).
- Вехи: Посев, Высадка в грунт, Первый урожай, Завершение сезона. **«Цветение» отложено** — стадии `flowering/harvesting` в приложении сейчас не выставляются, датировать нечем.

## 2. Навигация и структура экрана

Новая вкладка **«Профиль / Мой участок»** заменяет вкладку **«Инфо»** в `bottomNavItems` (`navigation/Navigation.kt`). Иконка профиля.

Экран — шапка + три таба:

- **Шапка профиля:** имя участка (`garden.name`), регион (климат-зона), счётчики: число фото · серия дней (из аналитики) · суммарный урожай (кг). Задел на будущее: аватар, соц-идентичность.
- **Таб «Лента»** (по умолчанию) — центральный контент (см. §3).
- **Таб «Статистика»** — существующий `AnalyticsScreen` (перенос точки входа из `InfoHubScreen`).
- **Таб «Справочник»** — объединяет «Справочник культур» (`CropsScreen`) и «Болезни и дефициты» (`GuideScreen`). Оба — reference-контент из бывшего `InfoHubScreen`; ни один вход не теряется.

`InfoHubScreen` расформировывается: его три карточки (Справочник культур, Болезни/дефициты, Статистика) переезжают в табы профиля.

## 3. Лента — модель и поведение

Хронология, **новые сверху**, с **разделителями по месяцам** (как в журнале действий). Элемент ленты — один из двух типов:

| Тип | Источник | Вид | Тап |
|-----|----------|-----|-----|
| `photo` | `planting_photos` | миниатюра + дата + культура + ярлык привязанного действия | полноэкранный просмотр (как в `PhotoViewerDialog`) |
| `milestone` | вычисляется (см. ниже) | авто-карточка с эмодзи + текст | экран той посадки (`PlantingInfo`) |

**Вехи (milestone) и их даты:**

| Веха | Источник даты |
|------|---------------|
| 🌱 Посев — `<культура>` | `plantings.planted_at` |
| 🌿 Высажено в грунт — `<культура>` | `logged_at` действия `transplanting` |
| 🌾 Первый урожай — `<культура> · N кг` | первая запись в `harvests` (`harvested_at`) |
| 🏆 Сезон завершён — `<культура>` | `plantings.updated_at` при `stage='done'` (приближённо; стадия терминальная) |

Привязка фото к действию (ярлык «Полив»/«Обработка») переиспользует уже реализованную в `PlantingInfoScreen` логику (`actionId → ACTION_LABELS[type]`).

## 4. Бэкенд

Новый эндпоинт **`GET /feed`** (авторизованный, пагинированный):

- Возвращает слитый хронологический список типизированных элементов: `{ type: 'photo'|'milestone', date, ...payload }`, сорт по дате убыв., пагинация (cursor или `limit/offset`).
- Собирается из существующих данных по владельцу: `planting_photos`, `harvests`, `plantings.planted_at`, action-логи (`action_type='transplanting'`), `plantings` со `stage='done'`. Изоляция по `user_id` через `gardens`.
- **Без миграции БД.**
- Это «шов» под будущую соц-ленту: позже здесь подключаются чужие ленты / фильтр видимости.

Существующий `GET /photos?planting_id=` остаётся для пер-посадочной ленты.

## 5. Задел под социализацию (не реализуется в v1)

- Форма элемента ленты стабильна и самодостаточна; будущие поля (`visibility`, `shared_at`, автор) добавляются к элементу/эндпоинту без переделки экрана.
- Серверная сборка ленты (а не клиентская) выбрана именно как точка расширения под чужие ленты.
- Никакого соц-кода, публичных URL, профилей других пользователей в v1.

## 6. Android — компоненты

- `navigation/Navigation.kt` — заменить `Screen.Info` на `Screen.Profile` в `bottomNavItems`; маршрут профиля.
- `ui/profile/ProfileScreen.kt` (новый) — шапка + табы (Лента / Статистика / Справочник).
- `ui/profile/FeedScreen.kt` + `FeedViewModel` (новый) — загрузка `/feed`, рендер ленты, пагинация.
- Общие компоненты ленты — вынести из `PlantingInfoScreen.kt` (`PhotoFeedRow`, `monthYearLabel`, формат даты) в общий файл `ui/feed/FeedItems.kt` и переиспользовать в обоих местах.
- `data/repository/FeedRepository.kt` + модель `FeedItem` (sealed: `Photo` / `Milestone`).
- `InfoHubScreen.kt` — удалить/расформировать; входы перенести в табы профиля.

## 7. Тестирование

- **Бэкенд (vitest, `npx vitest run`):** `/feed` — слияние типов, сортировка по дате, пагинация, изоляция по владельцу (чужие фото/вехи не попадают).
- **Android:** компиляция флейвора (`:app:compileRustoreDebugKotlin`); проверка состава ленты и переходов на тестовом аккаунте.

## 8. Вне объёма v1

Веха «Цветение» (нужна отметка стадии `flowering`) · шеринг поста · профили других пользователей · подписки/лайки/комментарии · публичные ссылки.
