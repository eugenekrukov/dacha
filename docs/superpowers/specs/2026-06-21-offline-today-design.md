# F1 — Офлайн-режим «Сегодня» (read-кэш + очередь записи)

**Дата:** 2026-06-21
**Платформы:** Android (A) + Backend (B). Веб вне scope (полевой сценарий — мобильное приложение).
**Статус:** дизайн утверждён, готов к плану реализации.

## 1. Проблема и цель

Дача часто без устойчивой связи. Главный экран «Сегодня» завязан на API (`/today`,
`/recommendations`, `/plantings`, `/actions`): при сетевой ошибке `TodayViewModel` уходит в
`TodayUiState.Error` — **кэша нет**, офлайн = экран ошибки. Это удар по основному сценарию ровно
там, где он нужен (инсайт №1 product-research-2026).

**Цель F1:**
- **Чтение офлайн:** последний успешный ответ «Сегодня» кэшируется и показывается без сети + баннер.
- **Запись офлайн:** действия (лог / смена стадии / удаление) кладутся в очередь и синхронизируются
  при возврате связи. Оптимистичный UI — действие видно сразу.

## 2. Подход

**A — лёгкий самописный кэш + очередь в репозиториях**, persisted через Moshi в
`SharedPreferences` (тот же стек, что `TokenStorage`). Без новых зависимостей (Room/WorkManager
отклонены: фоновая синхронизация при закрытом приложении не оправдывает вес — реальный сценарий
«поймал связь / открыл приложение» покрывается триггерами ниже).

**Конфликтная поверхность узкая:** `action_logs` — append-only журнал на пользователя; офлайн-записи
дописываются. Смена стадии идемпотентна. Единственный реальный конфликт — задвоение при ретрае,
закрывается клиентским ключом идемпотентности (`client_id`).

## 3. Backend (B) — миграция 045

- **`action_logs.client_id UUID` (nullable)** + частичный `UNIQUE` индекс
  `WHERE client_id IS NOT NULL`. UUID генерит клиент при постановке в очередь.
- **`POST /actions`** принимает опциональные `client_id` и `logged_at`:
  - `INSERT ... ON CONFLICT (client_id) DO NOTHING`; при конфликте — `SELECT` существующей строки и
    вернуть её (ретрай идемпотентен, тот же ответ/код).
  - `logged_at = COALESCE($client_logged_at, NOW())` — офлайн-действие ложится своим временем;
    онлайн без поля — `NOW()` как сейчас.
  - `client_id` валидируется как UUID; кривой → 400. Старые клиенты (без поля) работают по-прежнему.
- **`DELETE /actions/:id`** уже идемпотентен: удаление отсутствующей строки → 404; клиент на
  синхронизации трактует 404 как успех.
- Стадия «Высадки» меняется через существующий `PUT /plantings/:id` (idempotent set `transplanted`)
  — бэкенд не трогаем.
- **Тесты (vitest):** дубль с тем же `client_id` → одна строка; переданный `logged_at` сохраняется;
  кривой UUID → 400; путь без `client_id` не сломан.

## 4. Read-кэш «Сегодня» (Android)

Новый `TodayCache` (Singleton, отдельный `SharedPreferences`-файл `dacha_today_cache`, Moshi):

- **Запись.** После успешного `loadToday()` целиком сохраняем `TodayScreenData` (ответ `/today` с
  задачами и погодой, `recommendations`, `plantings`, `todayActions`) + `cachedAt` (epoch-millis) +
  `gardenId`. Кэш — на конкретный участок: при смене активного сада старый кэш не показываем.
- **Чтение.** В `loadToday()`: если `todayRepository.getToday()` падает **сетевой** ошибкой
  (`IOException`/`UnknownHost`/timeout) **и** есть валидный кэш для текущего `gardenId` — эмитим
  `TodayUiState.Success` с данными из кэша, флагом `offline=true` и `cachedAt`. Нет кэша → `Error`
  как сейчас.
- **Различаем ошибки.** Сетевую недоступность отделяем от прикладных (401/403/5xx). Тип ошибки
  пробрасываем из репозитория (расширим `Result.Error` полем `isNetwork` либо подтипом). Только
  сетевая включает офлайн-режим из кэша.
- **UI.** В `TodayScreenData` += `offline: Boolean`, `cachedAt: Long?`. Сверху экрана баннер
  «Нет связи · данные от ЧЧ:ММ» (стиль существующих баннеров). При восстановлении связи и успешной
  загрузке баннер исчезает.
- Клиентское dismiss/snooze-состояние задач/советов уже в `TokenStorage`, работает офлайн без правок.

## 5. Очередь записи (Android)

Новый `ActionQueue` (Singleton, persisted Moshi → `SharedPreferences` `dacha_action_queue`). FIFO,
каждая запись:

```
QueuedOp(
  clientId: String,        // UUID, генерится при enqueue — ключ идемпотентности
  op: LOG | STAGE | DELETE,
  plantingId: Int,
  // LOG:
  type: String?, notes: String?, auto: Boolean, loggedAt: String?,  // ISO, клиентское время
  // STAGE:
  stage: String?,          // "transplanted"
  // DELETE:
  targetServerId: Int?,    // id серверного действия
  targetClientId: String?, // если удаляем ещё не синхронизированный LOG из очереди
  createdAt: Long
)
```

**Запись действия** (правим `ActionsRepository`):
- Пытаемся отправить сразу, **всегда** передавая `clientId` + `loggedAt`. Успех → готово.
- Сетевая ошибка → enqueue, возвращаем `Result.Success` оптимистично. Прикладная ошибка (403 IDOR
  и пр.) → `Result.Error`, в очередь НЕ кладём.

**Удаление:**
- Цель ещё в очереди (`targetClientId`, не синхронизирована) → выкидываем тот `LOG` из очереди, на
  сервер ничего не уходит.
- Цель — серверное действие, офлайн → enqueue `DELETE` по `targetServerId`.

**Высадка** = две операции: `LOG(transplanting)` + `STAGE(transplanted)` для одной посадки.

**Синхронизация `syncQueue()`**, триггеры: возврат связи (`ConnectivityManager.NetworkCallback`),
foreground, успешный `loadToday()`. Шлём по порядку (FIFO), под `Mutex` (без параллельных прогонов):
- Успех / 404 на DELETE / конфликт по `client_id` → снимаем из очереди.
- Сетевая ошибка → стоп, остальное ждёт следующего триггера.
- Неретраибельная 4xx (403/400) → снимаем из очереди + лог (защита от вечного цикла).

**Индикатор:** число операций в очереди реактивно (StateFlow); на баннере «Сегодня» —
«N действий ждут отправки», после синхронизации сбрасывается.

## 6. Оптимистичное локальное состояние

Офлайн нельзя пересчитать задачи на сервере (`buildTasks` серверный) → оптимизм держим локально.

- `LOG` в очередь → (1) добавляем синтетическое действие в `todayActions` кэша (id по `clientId`,
  флаг `pending=true` → пометка «↑ ждёт отправки» в ленте/журнале); (2) локально «закрываем»
  соответствующую задачу на «Сегодня» через существующий механизм `deletedTasks`/pending в
  `TokenStorage` — задача уходит сразу.
- Синхронизация → действие получает реальный серверный id, `pending` снимается; следующий успешный
  `loadToday()` берёт серверную правду, синтетика заменяется реальными данными.
- Удаление pending-действия до синхронизации → убираем из очереди и из `todayActions`, задачу
  возвращаем (снимаем локальное «закрытие»).

## 7. Граничные случаи

- **Смена активного сада:** кэш и очередь привязаны к `gardenId`; очередь синхронизируем независимо
  от текущего сада (операции несут свой `plantingId`).
- **Logout:** очередь и кэш чистим (как push-токен) — чужие данные не показываем/не отправляем.
- **Переполнение:** мягкий потолок очереди ~200 операций.
- **Старый серверный ответ** без `client_id` в `todayActions` — мерж по серверному id, синтетика
  только для pending.

## 8. Тестирование

- **Backend (vitest):** дедуп по `client_id`, приём `logged_at`, кривой UUID → 400, обратная
  совместимость пути без `client_id`.
- **Android (unit):** `ActionQueue` (enqueue/dedup/FIFO/удаление pending), `TodayCache`
  (запись/чтение/привязка к саду), маппинг ошибок (сетевая vs прикладная).
- **Верификация:** компиляция флейворов (`:app:compileGplayDebugKotlin`) — инструментальные тесты в
  проекте не идут (баг AGP 9, см. session-note).
