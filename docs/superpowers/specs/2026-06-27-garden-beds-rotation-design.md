# Грядки участка + подсказка севооборота — дизайн

**Дата:** 2026-06-27
**Статус:** УТВЕРЖДЁН, к реализации.
**Базовый код:** `main` `7e29037`.

## Проблема

В справочнике проблем растений уже есть советы вида «севооборот минимум 3–4 года»
(вертициллёзное увядание, фузариоз и т.п.), но приложение не может их применить: посадка не привязана
к месту на участке, поэтому неизвестно, что росло на этом месте в прошлые годы. Полноценный визуальный
планировщик грядок (как у конкурентов — drag-and-drop карта участка) рассматривался, но отклонён как
слишком дорогой в разработке и редко используемый (раз в сезон, не каждый день) относительно
практической пользы. Решение — облегчённый вариант: грядка как именованное место + подсказка о
севообороте по ботаническому семейству, без визуальной карты.

## Решение (утверждено по разделам)

### 1. Данные

- Новая таблица **`garden_beds`**: `id, garden_id, name, type ('soil'|'greenhouse'), created_at`.
  Принадлежит участку — те же правила IDOR, что у прочих ресурсов (`garden_id ∈ user's gardens`).
- Новая колонка **`crops.family`** (text, nullable) — ботаническое семейство. Разовый backfill для
  всех 55 культур (контентная задача, список «культура → семейство» — отдельным шагом перед миграцией,
  на проверку владельцу до заливки в БД).
- Новая nullable колонка **`plantings.bed_id`** → `garden_beds.id` `ON DELETE SET NULL` (удаление
  грядки не ломает старые посадки, просто снимает привязку).
- `plantings.conditions` не меняется структурно: при выборе грядки в форме просто **подставляется**
  значение `bed.type` как стартовое — хранится как снимок на момент посадки (не живая ссылка), можно
  поменять руками после подстановки. Грядка без названия дублей не блокируется (uniqueness не вводим —
  пользователь сам увидит дубликат имени в списке при выборе).

### 2. API (бэкенд)

- `GET /gardens/:id/beds` — список грядок участка, **с встроенной историей посадок за последние
  3 года** на каждой грядке (название культуры, `family`, год посадки). Это закрывает всё, что нужно
  клиенту для подсказки севооборота за один запрос — без отдельного round-trip при каждой смене
  культуры в форме.
- `POST /gardens/:id/beds` `{ name, type }` — создать грядку (используется и из инлайн-«+ Новая
  грядка» в форме посадки).
- `PATCH /beds/:id` `{ name?, type? }`, `DELETE /beds/:id` — переименовать/удалить, с проверкой
  принадлежности участка пользователю (как у остальных ресурсов).
- `POST /plantings` и `PATCH /plantings/:id/info` — принимают опциональный `bed_id`.
- Сравнение «совпадает семейство за последние 3 года» считается **на клиенте** (история уже пришла
  целиком с грядкой) — на бэкенде отдельного эндпоинта проверки нет.
- Окно сравнения: `planted_at >= NOW() - INTERVAL '3 years'` (включительно).

### 3. UI-флоу (Android + Web, зеркально, обе платформы делаем сразу)

- В форме посадки — новое поле **«Место»** между «Культура» и «Условия» (порядок: причина выше
  следствия — выбор грядки определяет «Условия» под ним).
- Тап по полю открывает список грядок участка + пункт **«+ Новая грядка»** прямо в этом же оверлее
  (название + чипы Грунт/Теплица) — без отдельного экрана управления грядками.
- Переименовать/удалить грядку — в том же списке (долгое нажатие на Android / иконка рядом со
  строкой на вебе).
- Выбор грядки авто-подставляет «Условия» (Грунт/Теплица) под полем «Место» — можно поменять вручную
  после подстановки.
- Если семейство выбранной культуры совпадает с семейством в истории этой грядки за последние 3 года —
  под полем «Место» появляется мягкое предупреждение (не блокирует сохранение), например:
  «На грядке «Теплица 1» в 2025 рос паслёновый (Томат) — для этого семейства рекомендуют перерыв
  3–4 года».
- Поле полностью необязательное: старые посадки без «Места» работают как сейчас, предупреждений для
  них не показываем (выбор не сделан — истории сравнивать не с чем).
- На экране посадки (PlantingInfoScreen / PlantingDetailScreen) — «Место» можно посмотреть и
  поменять задним числом, как сорт и условия сейчас.

### 4. Контент — ботанические семейства

Разовая задача отдельным шагом: составить таблицу «культура → семейство» для всех 55 культур
(Паслёновые, Тыквенные, Крестоцветные, Бобовые, Луковые, Зонтичные, Маревые/Амарантовые, Астровые,
Злаковые, Розоцветные и т.п.), показать владельцу на проверку, затем — миграция backfill.

### 5. Тестирование и обработка ошибок

- Бэкенд (vitest): IDOR-тесты CRUD грядок (нельзя получить/изменить чужую грядку), корректность
  3-летнего окна истории на границе, создание/обновление посадки с `bed_id`.
- Удаление грядки с историей посадок — `ON DELETE SET NULL`, без отдельной обработки ошибок.
- Дубликаты названий грядок — не блокируются.
- Фронтенд (Android/веб) — у проекта нет сложившейся практики автотестов экранов/форм (только бэкенд
  на vitest); проверка вручную через сборку/превью, как и для прочих UI-фич в проекте.

  не обсуждали; по умолчанию — только в форме создания/редактирования, не на карточке списка посадок.

## Что НЕ делаем (явно отклонено)

Визуальная карта/планировщик участка (drag-and-drop расстановка культур по плану грядок) — отклонено
как слишком дорогое относительно частоты использования. Если эта потребность станет острее —
отдельная фича, отдельный дизайн.
