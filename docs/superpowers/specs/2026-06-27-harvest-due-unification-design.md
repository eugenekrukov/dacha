# Дизайн: унификация действия «Убрать урожай» (Android + веб)

## Проблема

Карточка задачи дня `harvest_due` («Убрать урожай: <культура>») генерируется backend'ом
(`backend/src/utils/todayLogic.js`), когда `daysSincePlanting >= crop.harvest_days`. Сейчас у неё
нет привязанного быстрого действия ни на одной платформе:

- На вебе `preselectFor()` (`web/src/screens/TodayScreen.tsx`) возвращает `null` для `harvest_due` —
  клик открывает форму действия без преднастройки.
- На Android клик падает в общий тип `"other"` (`TodayScreen.kt`) — то же самое, без смысла.

Отдельно в Android (но не на вебе) есть пункт «Завершить сезон» в списке посадок — переводит
`plantings.stage = 'done'` (архивирует посадку). На вебе такого пункта нет вообще.

Карточка `harvest_due` генерируется одинаково для всех культур и не исчезает сама — единственный
способ её остановить сейчас — завершить сезон. Для культур с однократным сбором (морковь, картофель)
это нормально, но для культур с многократным сбором за сезон (огурцы, малина, кабачки) механически
маппить клик по карточке на «завершить сезон» неправильно — посадка закроется преждевременно.

В системе логирования действий (`action_logs`, типы watering/fertilizing/treatment/other) отдельного
типа «урожай» нет. Однако в проекте уже существует независимый, готовый механизм логирования урожая:
таблица `harvests` (`planting_id`, `weight_kg`, `quantity`, `notes`, `harvested_at`) + `POST/GET /harvests`,
с готовым UI на обеих платформах (`HarvestScreen.kt` / `HarvestsScreen.tsx`) — независимый от
`plantings.stage`, поддерживает произвольное число записей на одну посадку. Используем его, а не
изобретаем новый механизм.

## Решение

### 1. Backend — cooldown для `harvest_due`

В `buildTasks()` (`backend/src/utils/todayLogic.js`) добавляется проверка: если по посадке есть
запись в `harvests` с `harvested_at` за последние **3 дня**, задача `harvest_due` для этой посадки
не генерируется в текущем проходе. Логика одинакова для всех культур — никакой классификации
«однократная/многократная» по справочнику культур не вводится (риск ошибиться в данных по 55+
культурам перевешивает пользу). Данные читаются одним запросом по всем активным посадкам участка,
по аналогии с уже существующими `lastWateredMap`/`lastFertilizedMap` в том же файле.

Никаких новых таблиц, полей или эндпоинтов: логирование — существующий `POST /harvests`,
завершение сезона — существующий `PATCH /plantings/:id/stage`.

### 2. Android

- `AddHarvestSheet` (используется в `HarvestScreen.kt`) получает:
  - опциональный параметр `preselectedPlantingId: Int?` — если задан, пикер посадки скрывается,
    форма открывается сразу с этой посадкой;
  - чекбокс **«Это весь урожай в этом сезоне»** — отображается всегда (и при обычном добавлении из
    журнала, и при открытии с карточки дня), по умолчанию выключен.
  - При сохранении: вызывается `addHarvest(...)` как сейчас; если чекбокс включён — сразу после
    успешного сохранения дополнительно вызывается `updateStage(plantingId, "done")`.
- Клик по карточке `harvest_due` на `TodayScreen.kt` открывает `AddHarvestSheet` с
  `preselectedPlantingId = task.plantingId` вместо текущего падения в `quickActionType = "other"`.
- Список посадок (`PlantingsScreen.kt`) не меняется — существующий пункт «Завершить сезон» остаётся.

### 3. Web

- Форма добавления урожая на `HarvestsScreen.tsx` получает тот же чекбокс «Это весь урожай в этом
  сезоне» и опциональный преселект посадки (та же логика, что в Android).
- `TodayScreen.tsx`: `preselectFor()` для `harvest_due` больше не возвращает `null`. Клик по карточке
  открывает форму урожая (модалка по аналогии с формой действия, переиспользующая компонент формы с
  `HarvestsScreen.tsx`) с преселектом посадки из `task.planting_id`.
- В список посадок на вебе добавляется пункт **«Завершить сезон»** (сейчас отсутствует полностью) —
  вызывает `PATCH /plantings/:id/stage` с тем же `stage: 'done'`, с диалогом подтверждения
  (как в Android: «Посадка будет переведена в архив. Данные сохранятся»).

### 4. Данные/миграции

Не требуются. Используются существующие таблицы `harvests` и поле `plantings.stage`.

### 5. Тестирование

- Backend (`backend/src/__tests__/`, vitest): тест на cooldown — посадка с harvest-записью младше
  3 дней не попадает в `harvest_due`; с записью старше 3 дней (или без записей вовсе) — попадает,
  как раньше. Плюс существующие тесты `buildTasks`/`harvests` не должны сломаться.
- Android/Web: ручная проверка флоу — клик по карточке → форма с преселектом → сохранение без
  чекбокса (карточка пропадает на 3 дня, посадка остаётся активной) → сохранение с чекбоксом
  (посадка переходит в архив). Отдельно — ручная проверка нового пункта «Завершить сезон» на вебе.

## Из дальнейшего
