# Дизайн: публичные SEO-страницы справочника (`/spravochnik/`)

**Дата:** 2026-07-01
**Статус:** утверждён владельцем, готов к написанию implementation-плана.

## Проблема / цель

Лендинг `dacha.studio1008.com` живёт только на брендовом трафике (люди, уже знающие
название) — в `sitemap.xml` всего 4 URL. При этом бэкенд уже отдаёт публично (без
авторизации) структурированные данные, которые закрывают длинный хвост
информационных запросов:

- `GET /crops` — 45+ культур (сроки посева/высадки, полив, морозостойкость,
  совместимость, заметки).
- `GET /guide` / `GET /guide/:slug` — 68 записей о дефицитах/болезнях/вредителях
  (симптомы, причины, лечение, профилактика, фото).

Цель — отрендерить эти данные как индексируемые HTML-страницы на том же домене,
с CTA в приложение, без переписывания уже существующего контента.

## Архитектура: статическая генерация (SSG), не SSR

Контент культур и записей справочника меняется редко — только через admin CRUD,
на практике всегда миграциями. Рендерить его на каждый HTTP-запрос смысла нет.

Скрипт `backend/scripts/generate-spravochnik.js` подключается напрямую к Postgres
(как остальные скрипты в `backend/scripts/`) и генерирует статические HTML-файлы
в `landing/spravochnik/`. Деплой — тем же способом, что и сам лендинг (см.
`docs/DEPLOY.md`, раздел «Лендинг (отдельно!)»): копирование каталога на VPS в
`/var/www/dacha-landing/spravochnik/` + новый `location /spravochnik/` в nginx.

Это не трогает Fastify API: ноль рантайм-нагрузки, страницы — обычная статика
(быстро, кэшируется), ноль риска для существующих JSON-роутов `/crops` и `/guide`
(они остаются как есть, для мобильного/веб-клиента).

Отклонённые альтернативы:
- **SSR на Fastify** — добавляет шаблонизацию и рендеринг в сервис, который должен
  оставаться тонким JSON API; не даёт выгоды, так как контент почти статичен.
- **Пререндер внутри `web/` (React SPA)** — `web/` сейчас авторизованное приложение
  за `/app/`; выделять публичные пререндеренные роуты внутри него непропорционально
  сложнее, чем отдельная генерация, и не даёт лёгких быстрых страниц.

## URL-структура

`/crops` и `/guide` уже заняты JSON API на том же домене → новый префикс `/spravochnik/`:

- `/spravochnik/` — хаб (ссылки на оба раздела)
- `/spravochnik/kultury/` — список культур, сгруппированный по категориям
  (овощи/зелень/ягоды/цветы)
- `/spravochnik/kultury/{slug}/` — страница культуры
- `/spravochnik/problemy/` — список проблем, сгруппированный по типу
  (дефициты/болезни/вредители)
- `/spravochnik/problemy/{slug}/` — страница проблемы

## Изменение схемы: `crops.slug`

`guide_entries.slug` уже существует. У `crops` slug нет. Новая миграция:

```sql
ALTER TABLE crops ADD COLUMN slug TEXT UNIQUE;
```

Бэкфилл — одноразовый скрипт, транслитерация из `name` (напр. «Томат» →
`tomat`), с проверкой уникальности (при коллизии — суффикс `-2` и т.п.). Slug
генерируется один раз и дальше не меняется — URL страниц культур должны быть
стабильными между перегенерациями (SEO: URL не должен «плавать»).

Если админ добавит новую культуру без slug — скрипт генерации при отсутствии
slug транслитерирует на лету и пишет предупреждение в консоль (не блокирует
генерацию остальных страниц).

## Шаблон страницы

Общие элементы (обе detail-страницы):
- Уникальные `<title>` / meta description по шаблону на основе полей записи.
- `canonical`, OG/Twitter-теги (переиспользуют паттерн лендинга).
- JSON-LD: `Article` (headline/description/publisher — тот же `Person`, что и на
  лендинге) + `BreadcrumbList` (Главная → Справочник → Культуры/Проблемы → Название).
- `H1` = название культуры/проблемы — естественно содержит «Календарь дачника» в
  контексте бренда через хлебные крошки и футер.
- CTA-блок в конце: «Добавить «{название}» в свой Календарь дачника →» → `/app/`.
- Общий вынесенный `landing/spravochnik/assets/style.css` (не инлайн — 113 файлов
  с одинаковыми ~9 КБ CSS инлайном нерационально; общий файл кэшируется браузером
  один раз на весь раздел).
- Общий футер с юридическими ссылками (как на лендинге).

### Страница культуры (`/spravochnik/kultury/{slug}/`)
- Сроки посева/высадки — `sowing_start_day`/`sowing_end_day` (день года) конвертируются
  в человекочитаемый диапазон дат («с 20 марта по 10 апреля»).
- `transplant_days`, `harvest_days`, `watering_freq_days`, `frost_sensitive`, `notes`.
- Совместимые культуры (`companion_crops`, id[]) — ссылки на их страницы.
- «Проблемы этой культуры» — записи `guide_entries` через `crop_guide_entries`,
  отфильтрованные по `crop_id` (тот же join, что в `GET /guide?crop_id=`),
  ссылки на `/spravochnik/problemy/{slug}`.

### Страница проблемы (`/spravochnik/problemy/{slug}/`)
- Бейдж типа (дефицит/болезнь/вредитель) + уровень опасности (`danger`).
- `description`, `symptoms`, `conditions`, `treatment`, `prevention`, `season`.
- **Изображения** (явно запрошено владельцем):
  - Основное фото — `guide_entries.image_url`, если задано.
  - Список «Поражает культуры» — для каждой культуры своя фотография из
    `crop_guide_entries.image_url` (специфичная для пары культура↔проблема:
    «недостаток калия выглядит по-разному у огурца и томата», см. миграцию 028),
    с fallback на основное фото записи, если для конкретной культуры своего фото нет.
  - Под каждым изображением — подпись из `image_credit` (атрибуция лицензии,
    обязательна по комментарию в миграции 028 — не может быть отброшена).
  - Записи без единого фото (ни общего, ни per-crop) — рендерятся без блока
    изображений, без плейсхолдеров-заглушек.

## Sitemap

Скрипт перегенерирует `sitemap.xml` целиком: 4 существующих URL (главная, offer,
privacy, account-deletion) + сгенерированные `/spravochnik/...` (хаб + 2 индекса +
~113 detail-страниц ≈ 117 новых). Общий объём далеку от лимита в 50 000 URL —
отдельный sitemap-индекс не нужен.

## Перелинковка с лендинга

В навигацию (`.nav-links`) и футер (`.store-row`) лендинга добавляется ссылка
«Справочник» → `/spravochnik/` — чтобы раздел был доступен с самой авторитетной
страницы сайта и попадал в обход поисковых роботов с главной.

## Деплой / nginx

Новый location-блок (по аналогии с существующими в `docs/DEPLOY.md`):

```nginx
location /spravochnik/ {
    root /var/www/dacha-landing;
    try_files $uri $uri/ =404;
}
```

Добавляется команда копирования каталога `spravochnik/` в раздел «Лендинг
(отдельно!)» `docs/DEPLOY.md`, рядом с существующим копированием `index.html`.

## Проверка / тестирование

Автотестов не требуется (генератор статики, не бизнес-логика приложения). Вместо
этого:
- Скрипт при генерации сам проверяет: у каждой culture/entry непустой slug и файл
  создан; все внутренние ссылки (`companion_crops`, «поражает культуры», «проблемы
  этой культуры») указывают на реально сгенерированные файлы — при несовпадении
  генерация падает с ошибкой (self-check, без внешних тестовых фреймворков).
- Ручная проверка в превью (`dacha-landing`, порт 8777, уже настроен в
  `.claude/launch.json`) — открыть по одной странице каждого типа (культура с
  companion_crops и без, проблема с фото и без) и свериться со снапшотом/скриншотом.

## Вне рамок этой итерации

- Автогенерация slug через admin API при создании новой культуры (`POST /crops`) —
  не трогаем; при необходимости генератор сам подставит slug на лету.
- Региональные/сезонные лендинги («посевной календарь на июль 2026») — отдельная
  будущая итерация, не входит сюда.
- Блог/автопостинг статей на домен — отклонено на этапе анализа в пользу этой
  фичи (данные уже структурированы, не требуют написания текста с нуля).
